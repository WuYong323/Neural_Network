## Q1：train.py 跑出来准确率只有 70%，哪里错了？

**日期**：2026-05-09

### 问题
按笔记目标应到 93%+，但实测只有 70% 多。

### 根因（按影响排序）
1. **mini-batch 切片对不齐**（最致命）
   ```python
   xb, yb = Xs[i:i+batch], Ys[i:batch]   # 第二个少了 i+
   ```
   `i=0` 时还正常，之后 `Ys[i:batch]` 全是空数组。结果每个 epoch 只有第一个 batch 真正更新参数，60000 样本 batch=1024 本应更新 58 次，实际只更新 1 次。

2. **lr 偏小**：`config.py` 里 `lr=0.001`，纯 SGD 推荐 0.1，小了 100 倍。

3. **batch 偏大**：1024 太大，更新次数少；笔记推荐 128。

4. **初始化用了 Xavier 而不是 He**：ReLU 网络应该用 `√(2/fan_in)`，否则前向信号方差衰减。

### 顺带发现的非致命 bug
- 日志文件每 epoch 用 `'w'` 打开 → 被反复覆盖，应改 `'a'`
- checkpoint 路径写成 `checkpoint/` 少了 s，与 `makedirs("checkpoints")` 不一致
- f-string 内引号嵌套（`f"{config["tag"]}"`）在 Python < 3.12 是语法错误
- CSV 表头字符串之间漏逗号，导致两列被隐式拼接

### 收获
**排查训练问题的顺序**：
1. 先排代码 bug（切片、路径、日志）
2. 再排数学 bug（gradient_check、初始 loss ≈ log(C)）
3. 最后才调超参

新手最容易把 ①② 误判成 ③，结果疯狂调参也救不回来。

## Q2：为什么 batch 改小，准确率就上去了？

**日期**：2026-05-10

### 问题
把 batch 从 1024 改到 128，其他没动，准确率明显上升。

### 核心解释：SGD 的"有效步长" ≈ lr × 总更新次数

每个 epoch 的更新次数 = `N / batch`。

| batch | 每 epoch 更新次数 | 25 epochs 共更新 |
|---|---|---|
| 1024 | 58 | 1450 |
| 128  | 469 | 11725 |

batch 缩小 8 倍 → 更新次数变 8 倍 → 相当于训练时间被自动延长 8 倍。

如果 lr 本来就偏小，大 batch 下根本走不动；改小 batch 实质是在用"更多的步数"弥补"每步太小"。

### 经验缩放法则
- **线性缩放**：batch 翻倍，lr 也翻倍
- **平方根缩放**：batch 翻倍，lr ×√2

直觉：batch 大 → 梯度估计更准 → 可以用更大 lr 安全前进；batch 小 → 噪声大 → lr 要降下来求稳。

### 小 batch 的隐形好处
梯度噪声有**隐式正则化**作用，更容易找到平坦解，泛化更好。但这是次要原因，主因始终是"更新次数变多"。

### 验证方法
保持 batch=1024，把 lr 从 0.001 改到 0.05~0.1，准确率应该接近小 batch 的结果。如果接近 → 验证了"有效步长"理论；如果不接近 → 还有正则化效应在起作用。

### 收获
- **超参不能单独调**：lr / batch / epochs 是耦合的，要看乘积
- **准确率上不去时，先怀疑"训得不够"，再怀疑"模型容量不够"**：90% 的情况只是更新次数不够，不是网络太小
- **养成"反向验证"的习惯**：用一个变量代偿另一个变量，能验证你对原理的理解是否正确

## Q3：为什么 micrograd 训练时 CPU 只有 25%，是没按批运行吗？

**日期**：2026-05-13

### 问题
跑上周的 numpy 版 MNIST 时，任务管理器里 `python.exe` 的 CPU 接近 **99%**；今天跑本周手写的 micrograd 版 `train_moons.py`，只有 **25%**。第一反应是"我是不是没按批训练？"

### 结论先行
不是批的问题。**25% 不是 bug，是 Python 标量运算 + GIL 的必然结果**。"按批"这条路在 micrograd 里根本走不通——它要配合**向量化**才有意义，而 micrograd 没有向量化。

### 根因

**① 25% 就是"单核打满 / 总逻辑核数"**

这台机器是 4 逻辑核，一个核打满 = 任务管理器显示 25%；8 线程机上同样情况显示 12.5%。看到这个整除数字，基本可以锁定"只用了一个核"。

| 逻辑核数 | 单核满载时总体 CPU % |
|---|---|
| 4 | **25%** |
| 8 | 12.5% |
| 16 | 6.3% |

**② 单核独占的根因：Python GIL（全局解释器锁）**

CPython 有一把大锁，保证同一时刻一个 Python 进程里**只有一个线程**在执行字节码。于是：

- `threading.Thread` 不能加速 CPU 密集的 Python 代码（只对 I/O 等待有用）
- 想用多核，要么**在 C 扩展里释放 GIL**（numpy / PyTorch 走的路），要么**开多进程**

micrograd 的每一次 `a * b` 都是纯 Python：创建 `Value` 对象、构造 `_backward` 闭包、更新 `_prev`。全程卡在 GIL 里，其它核在等锁摸鱼。

**③ numpy 为什么能 99%？**

numpy 的核心运算链接到 BLAS（MKL / OpenBLAS），底层是 C/Fortran + SIMD + 汇编。一次 `X @ W`：

1. 进 C 代码前 **显式释放 GIL**（`Py_BEGIN_ALLOW_THREADS`）
2. BLAS 内部**自己开多线程**切块并行

一次 `X @ W`（batch=64、784→256）= 12M 次乘加在**一次 BLAS 调用里**完成，Python 只出场一次——这才是多核同时飙的原因。

**④ "按批"为什么救不了 micrograd？**

把 100 样本拆 10 batch，只是换了 for 循环结构，**每次 `Value * Value` 仍然是标量运算**。批在 numpy 里之所以能加速，是因为它把多个样本**打包成矩阵**交给 BLAS 一次算完。micrograd 里每个 Value 都是独立标量对象，没有"打包成矩阵"这一步，批在这里失去了加速意义。

因果链（记死）：

> 批加速 ← 向量化 ← 大块 C 调用 ← 释放 GIL + SIMD + 多线程 = CPU 打满

少了"向量化"这一环，光喊"按批"没用。

### 验证方法（三选一，别只看总体 %）

**① 逻辑核曲线图**（最直观）：任务管理器 → 性能 → CPU → 右键图形 → "逻辑处理器"。numpy 时多核同时飙；micrograd 时**只有一个核贴顶**。

**② psutil 看每核**：

```python
import psutil, os
p = psutil.Process(os.getpid())
print(p.cpu_percent(interval=1.0, percpu=True))
```

micrograd 期间大概 `[100, 0, 0, 0]`，numpy 期间 `[80, 70, 85, 75]` 左右。

**③ 等价实现对比**：把 `Value` 换 `ndarray` 重写一遍，时间掉一个数量级，CPU 同时跳到 90%+。

### 升级路径

| 阶段 | 数据结构 | 底层 | CPU 利用率 | 相对时间 |
|---|---|---|---|---|
| micrograd | 标量 `Value` | 纯 Python | ~25%（单核） | 1.0x |
| numpy 手写 MLP | `ndarray` | BLAS | 90%+（多核） | 0.01–0.05x |
| PyTorch CPU | `Tensor` | ATen + MKLDNN | 90%+（多核） | 0.01–0.05x |
| PyTorch GPU | `Tensor.cuda()` | cuBLAS / cuDNN | CPU 闲 GPU 忙 | 0.001x 或更低 |

Week4 要做的就是从第一格跨到第二格——向量化。

### 收获
- **25% = Python GIL × 标量运算；99% = BLAS × 向量化**——两边不是同一条优化曲线，是两个世界
- **"按批"不是加速咒语**，真正的加速是"一次调用覆盖大量乘加"；没有向量化，按批只是换循环写法
- **multi-threading ≠ multi-core**：Python 里 `threading` 对 CPU 密集任务无效，只有释放 GIL 的 C 扩展或多进程才能真正用多核
- **看 CPU 利用率要看每核曲线**，总体百分比在多核机上会骗你——特别是那些刚好等于 `100% / 核数` 的整除数字
- micrograd 慢且用不上多核**不是 bug，是它的定位**：负责讲清楚"自动微分怎么工作"，不负责跑得快。性能是 numpy/PyTorch 的职责

## Q4：train_moons.py 的 loss 不降反升，最后 tanh 里 exp 溢出

**日期**：2026-05-13

### 问题
修好 `zero_grad` 的拼写 bug（`p.grade → p.grad`）后，loss 仍然越训越大，跑到中途 `engine.py` 的 `math.exp(2*x)` 抛 `OverflowError`。第一反应是"tanh 的数值稳定性问题，要做 clip 或换写法"——错了。

### 结论先行
**`exp` 溢出不是病，是症状**。病是**"线性输出头 + MSE + lr=1.0 + 全批量"这四个条件一起构成了一个正反馈发散回路**。tanh 溢出只是回路跑到崩溃边缘时第一个尖叫的零件。不修回路、只去堵 tanh，等价于"温度计爆了就换个耐高温温度计"——治标不治本。

### 发散回路的数学推导（核心）

设输出层某个参数为 $w$，第 $i$ 个样本的线性 score 为 $s_i = w \cdot h_i + \ldots$，MSE 损失：

$$
L = \frac{1}{N}\sum_i (s_i - y_i)^2
$$

对 $w$ 的梯度：

$$
\frac{\partial L}{\partial w} = \frac{2}{N}\sum_i (s_i - y_i)\, h_i
$$

SGD 一次更新：

$$
w \leftarrow w - \eta \cdot \frac{2}{N}\sum_i (s_i - y_i)\, h_i
$$

**关键观察**：梯度幅度与残差 $(s_i - y_i)$ **线性成正比**，没有上界。这意味着如果某次更新把 $s_i$ 推得离 $y_i$ 更远，下一步梯度**就变得更大**，步长再变更大——正反馈。

把这一步带入下一轮的 score 变化：

$$
\Delta s \approx h \cdot \Delta w = -\eta \cdot \frac{2}{N} \|h\|^2 \cdot (s - y)
$$

令 $\alpha = \eta \cdot \frac{2}{N} \|h\|^2$，则 $s_{t+1} - y = (1-\alpha)(s_t - y)$。发散判据：

$$
|1-\alpha| > 1 \;\Longleftrightarrow\; \alpha > 2 \;\Longleftrightarrow\; \eta > \frac{N}{\|h\|^2}
$$

用当前配置估一下：`lr=1.0`、$N=100$、隐层 16 维 tanh 输出 $\|h\|^2 \approx 16$，阈值约 $\eta_{crit} \approx 6$。看起来 1.0 还很安全？**不对**——上面只估了输出层一个权重。反向传到隐层时，梯度会再乘一遍 $w$（当前输出权重），而初始化 `random.uniform(-1,1)` 下 $\|w\|^2$ 是 $O(16)$，所以隐层参数看到的有效步长被再放大一个数量级。**实际 lr=1.0 在这套配置下已经越过多层的联合稳定边界**。

> 记这个脑图：**$\eta$ 的安全上界不是常数，而是 $O\left(\frac{N}{\text{层数} \cdot \text{每层宽度}}\right)$**。

### 四个条件如何合谋

1. **输出层 `nonlin=False`**（`nn.py:49` 的三元判断，"最后一层线性"是通用约定）
   - 回归/分类 logits 通常希望无界，所以最后一层不加激活
   - 代价：`score` 没有被压缩，可以任意漂离 $y \in \{-1, +1\}$

2. **MSE 损失**
   - 梯度 $\propto (s-y)$，偏离目标越远梯度越大
   - 对比：hinge、cross-entropy、huber 都有"越远梯度越饱和或有界"的性质

3. **lr=1.0（且只线性衰减到 0.1）**
   - `1.0 - 0.9*epoch/EPOCH` 最小还是 0.1，前 90% 的训练都在 lr ≥ 0.19
   - 对纯 SGD + 无归一化的小网络，0.05 量级才是保险的起点

4. **全批量 GD（100 样本一起更新）**
   - 梯度方差小 = 没有噪声去打乱发散方向
   - mini-batch SGD 的噪声有时反而能把网络踢出发散轨迹

任意**去掉一条**都可能让回路不成立。Karpathy 原版 demo 能稳定用 lr=1.0，是因为它配的是 **hinge loss + ReLU**：

$$
L_i = \max(0,\; 1 - y_i \cdot s_i)
$$

hinge 在分类正确且置信度 ≥ 1（即 $y_i s_i \ge 1$）时梯度**精确为 0**。这意味着：一旦样本分对，它就退出训练，不再贡献梯度——天然抑制了"越训越偏"的正反馈。MSE 对已经分对的样本仍然在"推向精确等于 $y$"，对分类任务是过度优化。

### 为什么 tanh 是第一个崩的

发散路径：
1. 输出层 $s$ 越推越大
2. 反向传回隐层时，梯度也同比例放大
3. 隐层权重 $w$ 被大步更新 → 隐层 **pre-activation**（tanh 的输入）$z = w \cdot x + b$ 开始越过 ±100、±500
4. `engine.py:37` 的 $(e^{2z}-1)/(e^{2z}+1)$：$z > 354$ 时 $e^{2z}$ 超过 `float64` 上限 $\approx 1.8 \times 10^{308}$，抛 `OverflowError`

注意 **`math.tanh(x)` 本身永远不会溢出**（C 标准库对 $|x|$ 大时直接返回 ±1.0）。这里溢出是因为 micrograd 手写的 tanh 用了 `math.exp`，没有考虑数值饱和。换用 `math.tanh` 能**掩盖**症状——loss 会止住发散、稳定在一个很大的值——但这本质上等于**偷偷给梯度做了 clip**（$\tanh \to \pm 1$ 后局部导数 $1-\tanh^2 \to 0$），并没有修发散回路。

### 怎么修（按优先级 + 每种的代价）

**A. 只调 lr（最小改动，推荐先试）**
```python
lr = 0.05 * (1.0 - 0.9 * epoch / EPOCH)
```
- 代价：几乎为 0
- 预期：loss 单调下降到 <0.1，acc→100%

**B. 换 hinge loss（贴合 micrograd demo 的精神）**
```python
data_losses = [(1 + score * (-float(yi))).relu() for score, yi in zip(scores, y)]
```
- 注意 `yi` 是 `np.int64`，放左边会走 numpy 的 `__mul__` 路径搞出怪东西。**先 `float(yi)` 或把 score 放左边**——这是 micrograd 和 numpy 混用时的经典坑
- 代价：要加 ReLU 支持（你已经有）
- 预期：lr=1.0 可用，收敛更快

**C. 梯度裁剪**（治本但多加一层复杂度）
```python
max_norm = 5.0
g = sum(p.grad ** 2 for p in model.parameters()) ** 0.5
if g > max_norm:
    for p in model.parameters():
        p.grad *= max_norm / g
```
- 代价：多遍历一次参数
- 预期：配 MSE + lr=1.0 也能稳住；但本质上是在掩盖"lr 选得不对"

**D. tanh 换 math.tanh**（伪解，不该单独做）
- 单独做：loss 会从 NaN 变成一个很大的常数，训练看似"不崩了"但也没在学
- 和 A/B/C 一起做：是正确的数值安全加固
- **永远不要用"看起来不报错了"作为修复成功的标准**

### 通用的调试方法论（最重要）

这次踩的不是 bug，是**诊断顺序**：看到 `OverflowError` 第一反应改 tanh，是**用症状的位置定位病因**，而症状位置几乎从来不是病因位置。应该养成的反射：

**看到数值爆炸时的诊断阶梯**（从上往下，依次排除）：

1. **是谁先炸？** 不是谁报错谁有罪。tanh 报错可能是上游线性层的参数已经失控。打印 `max(abs(p.data))` for all params，先锁定真正膨胀的那层
2. **是输入还是参数炸？** 打印 `max(abs(h))` for 每层 pre/post activation；如果参数稳定但激活在炸，可能是某个样本异常；如果参数在炸，回路是训练本身
3. **loss 是一直涨还是先降后涨？** 一直涨 → 初始化 + lr 就错了；先降后涨 → 学过头、步长不匹配
4. **关掉正则项还炸吗？** 正则项梯度也会跟参数一起爆炸（$\partial (\alpha w^2)/\partial w = 2\alpha w$），有时反而是正反馈放大器
5. **lr 砍 10 倍还炸吗？** 如果砍到 0.01 还炸 → 不是 lr 问题，是损失函数或输出层结构错配；如果砍完就稳 → lr 就是主因

这套流程能把你从"改 tanh 修 exp 溢出"这种细枝末节上拉回到真正的矛盾。

### 更深一层：发散 vs 收敛 vs 震荡

把训练动力学想成一个一维映射 $s_{t+1} = (1-\alpha)(s_t - y) + y$：

| $\alpha$ 范围 | 行为 | 现象 |
|---|---|---|
| $0 < \alpha < 1$ | 单调收敛 | loss 平滑下降 |
| $\alpha = 1$ | 一步到位 | 理论最优（只在线性回归纯凸情形） |
| $1 < \alpha < 2$ | 震荡收敛 | loss 锯齿下降 |
| $\alpha = 2$ | 等幅震荡 | loss 在两个值之间跳 |
| $\alpha > 2$ | 发散 | loss 指数爆炸 ← 今天的情况 |

**lr 调参的目标不是找"最快的 lr"，而是找"$\alpha$ 刚好略小于 1 的 lr"**——让每一步都踏实靠近最优，又不浪费步数。这也是 lr scheduler 普遍"前期稍大、后期变小"的原因：前期地形陡（$\|h\|^2$ 大），配小 lr 压住稳定性；后期参数稳定、$\|h\|^2$ 下降，可以放开步长。不过实际训练里常反过来做（warmup + decay），因为前期"逃离坏初始化"的收益通常压过稳定性风险。

### 收获（背下来）

- **数值爆炸的报错位置几乎从不是病因位置**——症状是"最先崩的那个环节"，病因往往在它的上游几步
- **lr 的安全上界由损失函数 + 网络规模共同决定**，没有"通用的 lr=0.01"。记公式：$\eta_{safe} \sim O(N / (\text{层数} \cdot \text{宽度}))$
- **lr=1.0 能不能用，看损失函数是否自带"正确样本零梯度"的性质**（hinge、带 margin 的 loss）；没有这个性质时配 MSE 就是雷
- **MSE 对分类任务是过度优化**：对已经分对的样本继续施力，容易反而把分类决策搞坏，cross-entropy / hinge 是更合适的选择
- **修 bug 前先问"修完会发生什么？"**：如果答案是"报错换一种形式出现"或"看起来不报错但指标也没改善"，那就不是修复，是粉刷
- **全批量梯度下降在小网络上比 mini-batch 更容易发散**——少了噪声这层缓冲，回路更容易成环
- **"最后一层线性"是个好默认，但它把稳定性责任交给了损失函数**，选 loss 时要意识到这一点
- **数值稳定写法（`math.tanh` 替换 `exp` 手写版）是必要的加固**，但不是修发散的手段——加固和修 bug 是两件事，一次只做一件
- **把训练看作动力学系统**：一维映射 $s_{t+1} = f(s_t)$ 的收敛/发散/震荡三态对所有 DL 训练都适用，不止 MSE + 线性输出这一例

## Q5：神经元、层、MLP 的本质是什么？为什么能直接复用 Value 的反向传播？

**日期**：2026-05-13

### 问题
在 W3 Day4 的自测题里反复出现的一个概念题：用一句话说清楚 **Neuron / Layer / MLP** 各是什么、它们为什么能直接复用 Day2 写好的 `Value.backward()` 而不需要再写一行求导代码。这是看似简单实则决定后面看不看得懂 PyTorch 的题。

### 结论先行
- **Neuron** = 一个**带可学习参数的标量函数** $\mathbb{R}^n \to \mathbb{R}$，本体是仿射变换 + 非线性激活
- **Layer** = $n_{out}$ 个**共享同一输入向量**的 Neuron 并列，相当于一个标量到向量的"广播器"
- **MLP** = 多个 Layer **首尾相接**的复合函数 $f_L \circ f_{L-1} \circ \cdots \circ f_1$
- **能复用反向传播是因为整个 MLP 从头到尾只用了 `+ * tanh ** ReLU`——这些原子运算 Value 已经全部写过 `_backward`**。链式法则不需要在网络层重新发明，它在 `_backward` 闭包里就已经内建了

一句话压缩：**网络结构是参数的"组织方式"，求导是底层 Value 已经帮你做完的事**。这正是 PyTorch 的核心哲学——`nn.Module` 只组装计算图，`autograd` 只管反向。

### 三层抽象的数学定义

**① Neuron（神经元）**

$$
y = \phi\left(\sum_{i=1}^{n} w_i x_i + b\right) = \phi(\mathbf{w}^\top \mathbf{x} + b)
$$

- **组成**：$n$ 个权重 $\{w_i\}$ + 1 个偏置 $b$ + 一个非线性 $\phi$（tanh / ReLU / sigmoid）
- **接口**：吃 $n$ 维向量，吐 1 个标量
- **可学习参数数量**：$n + 1$
- **本质**：在 $n$ 维空间里画一个**超平面**（$\mathbf{w}^\top \mathbf{x} + b = 0$），$\phi$ 把"到超平面的有符号距离"非线性地压成激活值

**② Layer（层）**

$$
\mathbf{y} = \begin{bmatrix} y_1 \\ y_2 \\ \vdots \\ y_{n_{out}} \end{bmatrix}, \quad y_j = \phi(\mathbf{w}_j^\top \mathbf{x} + b_j)
$$

或写成矩阵形式（这就是 numpy/PyTorch 的高速路径）：

$$
\mathbf{y} = \phi(W\mathbf{x} + \mathbf{b}), \quad W \in \mathbb{R}^{n_{out} \times n_{in}},\ \mathbf{b} \in \mathbb{R}^{n_{out}}
$$

- **组成**：$n_{out}$ 个 Neuron，**全部接收同一个输入向量 $\mathbf{x}$**，但**各自有独立的 $\mathbf{w}_j, b_j$**
- **接口**：$\mathbb{R}^{n_{in}} \to \mathbb{R}^{n_{out}}$
- **可学习参数数量**：$n_{out}(n_{in} + 1)$
- **本质**：**把输入向量同时投影到 $n_{out}$ 个不同方向上**——每个 Neuron 关心一个不同的"特征方向"

注意 micrograd 实现里 Layer 没有 $W$ 矩阵，是一个 Neuron 列表 + 一个 for 循环。**数学上等价，工程上是 §Q3 性能问题的根源**。

**③ MLP（多层感知机）**

$$
f(\mathbf{x}) = f_L \circ f_{L-1} \circ \cdots \circ f_1 (\mathbf{x})
$$

每个 $f_\ell$ 是一个 Layer，**前一层的输出当下一层的输入**。

- **接口**：$\mathbb{R}^{n_{in}} \to \mathbb{R}^{n_{out}^{(L)}}$
- **可学习参数数量**：$\sum_\ell n_{out}^{(\ell)}(n_{in}^{(\ell)} + 1)$
- **本质**：**复合函数**——这正是为什么深度有意义、为什么会有梯度消失/爆炸、为什么链式法则是 DL 的核心定理

> 三层抽象的关系是**严格的"包含 = 复合"关系**：MLP 包含 Layer 是因为 MLP **复合** Layer；Layer 包含 Neuron 是因为 Layer **并列** Neuron。"包含"不是简单的代码组织——它对应了完全不同的数学操作。

### 几何直觉（让"为什么需要多层"变得显然）

**单个 Neuron 的几何**：在输入空间里切一刀。线性切——超平面；非线性切——一个 S 形曲面（tanh）或一个折角（ReLU）。**单 Neuron 只能做"线性可分"的二分类**。

**单层（无激活）**：对输入做仿射变换，几何上是**旋转 + 缩放 + 平移**。无论叠多少层，**线性的复合还是线性**：$W_2(W_1\mathbf{x}+\mathbf{b}_1)+\mathbf{b}_2 = (W_2W_1)\mathbf{x} + (W_2\mathbf{b}_1+\mathbf{b}_2)$，等价于一个单层。**这就是为什么必须有非线性激活**——没有它，深度无意义。

**单层（带激活）**：每个 Neuron 切一刀，整层就是 $n_{out}$ 个**独立的切刀**作用在同一个空间。仍然只能描述凸的、被半空间合成的区域。

**两层（带激活）**：第二层的输入是第一层切完的"激活空间"——**第一层把空间扭曲了，第二层在扭曲后的空间里再切**。这就够把 make_moons 的两个半月分开了。

**三层及以上**：可以理解为**"反复扭曲再切"**，每次扭曲都让原本线性不可分的问题在新空间里变得线性可分（这正是 kernel trick 的精神，但 kernel 是固定的，神经网络是**学出来的扭曲**）。

> 一句话：**深度 = 学到的非线性坐标变换的次数**。这是 representation learning 的几何根。

### 为什么能复用 Value 的反向传播（核心机制）

这是这道题最关键的部分。回到 Day2 的 `Value` 类——它做了**两件事**：

1. **正向**：每次运算都创建一个新 `Value`，并记录 `_prev`（父节点）和 `op`（操作类型）
2. **反向**：每个运算都附带一个 `_backward` 闭包，知道"我这一步的局部导数怎么传回上游"

举具体例子：你写

```python
y = (w1*x1 + w2*x2 + b).tanh()
```

实际上 Value 在内存里建的是这样一棵图（DAG）：

```
        x1   w1     x2   w2
         \   /       \   /
        m1 = w1*x1   m2 = w2*x2
              \       /
               s1 = m1 + m2
                 \      \
                  s2 = s1 + b
                       |
                  y = s2.tanh()
```

**关键事实**：
1. 这棵图里**每条边都有一个已知的局部导数**——因为 `*`、`+`、`tanh` 的 `_backward` 你都写过
2. `y.backward()` 做的事就是**从 `y` 出发反向 DFS，把每条边的局部导数累乘起来**——这就是链式法则的代码化身
3. **图的形状是什么、有多深、有多宽，对 `backward` 的代码完全没影响**——它只关心"父子关系 + 每条边的局部导数"

**所以**当你写一个 MLP：

```python
y = model(x)  # 内部展开成几千次 Value 的 + * tanh
loss = (y - target)**2
loss.backward()
```

Value 看到的是一棵**几千个节点的 DAG**，但每个节点的 `_backward` 处理的还是它原来熟悉的"一次乘法"或"一次加法"或"一次 tanh"。**链式法则的"链"长一点、短一点、有分叉、有共享，都不重要**——递归本身能处理任意拓扑。

> **复用的根本理由**：链式法则是关于**两条边相邻**的局部规则。Value 在每条边上写好了这个规则，整张图的反向传播就是这个规则**自动累积**的结果。MLP 没有给 Value 增加任何新原子运算——它只是用现有原子搭了一个更大的图。

### 反过来验证：**如果**新增了一个原子怎么办？

假设你想加 sigmoid。Value 没有这个方法，于是你必须：

1. 在 `engine.py` 里写 `def sigmoid(self):`
2. 给它写局部导数：$\sigma'(x) = \sigma(x)(1-\sigma(x))$
3. 在 `_backward` 里把这个导数乘上 `out.grad` 累加回 `self.grad`

**但 nn.py 里 Layer / MLP 一行不用改**——因为它们只组装运算，不实现求导。这是一个**强对称性**的证据：求导逻辑只在原子运算层，组装层永远不需要懂求导。

PyTorch 也是这个对称性：你写新的 `nn.Module` 不用写求导代码（除非你定义新的 `autograd.Function`，那相当于在 micrograd 里加新原子）。

### 三层抽象 → PyTorch 的精确对应

| micrograd 概念 | PyTorch 概念 | 对应文件 |
|---|---|---|
| `Value` 类 + `_backward` 闭包 | `Tensor` + `grad_fn` | `torch.autograd` |
| `engine.py`（求导原子） | `aten/`（C++ 算子）+ 自动生成的反向 | `torch._C` |
| `Module` 基类 + `parameters()` | `nn.Module` + `.parameters()` | `torch.nn.Module` |
| `Neuron` / `Layer` / `MLP` | `nn.Linear` + `nn.Sequential` | `torch.nn` |
| `model.zero_grad()` | `optimizer.zero_grad()` | `torch.optim` |
| `for p: p.data -= lr*p.grad` | `optimizer.step()` | `torch.optim.SGD` |

**今天手搓一遍 micrograd → 明天看 PyTorch 源码不会慌**。这种"手做最小可工作版本"再去读真实框架的方法，比直接读 PyTorch 文档省 10 倍时间。

### 几个容易混淆的细节

**① 为什么 Layer 不直接是矩阵乘法？**

数学上 `Layer(x)` 完全等价于 `phi(W @ x + b)`。micrograd 没用矩阵是因为 `Value` 是标量类型，矩阵化要切到 numpy；而切到 numpy 就失去了"逐节点求导"的教学意义。**等价性**和**实现方式**是两件事——上层数学等价，下层实现差几个数量级（见 Q3）。

**② 为什么"最后一层 nonlin=False"？**

最后一层的输出是 logits（分类）或预测值（回归），通常希望它**无界**。把 tanh 套上去会把输出压在 [-1, 1]——回归任务直接废，分类任务损失函数也得跟着变（cross-entropy 期待无界 logits）。这是 §Q4 发散问题的源头之一：线性输出层 + MSE 是个潜在的雷区。

**③ 共享参数 vs 独立参数**

Layer 里的 $n_{out}$ 个 Neuron **不共享参数**（各自的 $\mathbf{w}_j, b_j$）。但同一个 Neuron 在跨 batch、跨样本时**共享自己的参数**——同一个 $\mathbf{w}$ 服务 100 个样本。`Value.backward()` 处理共享靠的是 `self.grad += ...` 的累加语义：同一个 Value 在图里被多个下游节点用到时，它的梯度自然累加。**这就是 zero_grad 必须存在的根本原因**——不清零，跨 batch / 跨 epoch 的梯度也会无差别累加。

**④ 为什么 `Module.parameters()` 必须 yield 出**所有**参数？**

因为优化器要按列表更新；因为 zero_grad 要按列表清零；因为正则项 $\sum p^2$ 要按列表加。**`parameters()` 是组装层和优化层之间唯一的接口**。Layer 的 `parameters()` 把 Neuron 的展开，MLP 的 `parameters()` 把 Layer 的展开——这就是嵌套展平。

### 收获（背下来）

- **三层抽象 = 三种数学操作**：Neuron 是仿射 + 激活；Layer 是 Neuron 的**并列**；MLP 是 Layer 的**复合**。三种操作各对应不同的数学结构（标量函数 / 向量函数 / 复合函数），不是简单的代码组织
- **复用 Value 反向传播的根本原因**：链式法则是**两条相邻边的局部规则**，Value 在原子运算上写好这个规则后，**任意拓扑的图都能自动应用**——网络层不需要重写一行求导代码
- **新增一个原子要改 engine，新增一个网络结构不用改 engine**——这条对称性是 micrograd → PyTorch → JAX 共同的设计哲学
- **深度 ≠ 高级**：深度的真正意义是"学到的非线性坐标变换次数"，没有非线性激活时深度等价于单层
- **"最后一层线性"是约定**，因为下游 loss 通常期待无界输出；这把数值稳定的责任交给了 loss，所以 loss 选错（如分类用 MSE）就会暴雷
- **梯度的累加语义 = 参数共享的自然处理**：同一参数被多次使用时 grad 自动累加，这就是 zero_grad 必须存在的根本原因
- **"等价"和"实现"是两件事**：micrograd 的 Layer 和 numpy 的矩阵乘法在数学上等价，但实现差 100 倍；理解等价性帮你看懂教学版，理解实现帮你写出生产版
- **`Module.parameters()` 是组装层和优化层的唯一接口**——优化器、正则、zero_grad 全靠它。设计自己的 nn 框架时这是不可省的方法
- 看 PyTorch 源码前先做一遍 micrograd——`Tensor` 对应 `Value`、`grad_fn` 对应 `_backward`、`nn.Module` 对应 `Module`、`optimizer.step()` 对应 `for p: p.data -= lr*p.grad`，几乎一一对应

## Q6：为什么 micrograd 比 numpy 网络慢大约 100 倍？是因为 Python 单线程吗？

**日期**：2026-05-13

### 问题
上周写的 numpy 版 MLP 跑 MNIST，一次 forward + backward 是毫秒级；今天 micrograd 跑 `MLP(2,[16,16,1])` 对 100 个样本一个 epoch 是几百毫秒级。粗估单位算力差 **~100 倍**。第一反应："是不是因为 Python 单线程？"——**不是主因**。单线程只解释"为什么用不上多核"（Q3 已答），不解释"为什么单核也慢 100 倍"。这两件事必须分开看。

### 结论先行
**100 倍慢的根因是"每次运算的常数开销太大"，和并发无关**。主因按影响排序：

1. **Python 对象模型**（主力）：每次 `a*b` 都创建新对象 + 闭包 + 维护图结构，数十条字节码
2. **解释器开销**：Python 字节码解释、动态分派、属性查找——每次都要跑一遍
3. **失去向量化**：numpy 一次 `X @ W` 打包几百万次乘加进一个 C 调用，micrograd 一次只做一个标量乘法

**单线程 / GIL 是独立的另一轴**：就算 Python 能完美多核并行，每个乘法还是要跑几十条字节码，依然比 C 慢 100 倍。GIL 决定 CPU 利用率（25% vs 99%），**不决定单次运算的速度**。

一句话记死：**100 倍慢 = 常数开销大（对象 + 解释器）；25% CPU = 这些运算挤在一个核（GIL）**。两件事，两个维度。

### 一次乘法到底发生了什么（对比 numpy）

**numpy 版的一次 `X @ W`**（比如 `batch=64, 784→256`）：
- 一次 Python 函数调用，进入 `numpy.matmul`
- 进入 C 代码前**显式释放 GIL**
- BLAS（MKL / OpenBLAS）接管：SIMD 向量指令 + 多线程切块
- 内部循环跑完 `64 × 784 × 256 ≈ 12M` 次乘加
- **Python 只出场一次**，C 代码里跑的时候解释器不存在

**micrograd 版的一次 `Value * Value`**：
- 一次 Python 函数调用 `Value.__mul__`
- 参数检查：`isinstance(other, Value)`
- 一次 `float * float` 的标量乘法（真正的计算，只要 1 条 CPU 指令）
- 创建新 `Value` 对象：Python 对象分配 + 引用计数初始化 + 哈希表写入
- 构造 `_backward` 闭包：函数对象 + 闭包捕获作用域
- 构造 `_prev` 集合：Python set 分配 + 两次插入
- 拼接 `op` 字符串
- 返回新对象

**同一件事——一个 float 乘法——在 numpy 里是 1 条 CPU 指令的 1/12M，在 micrograd 里是几十条 Python 字节码的全部**。

这才是 100 倍的来源。和线程数、核数无关。

### 精确拆解：100 倍来自哪里

按贡献度粗估（具体数字取决于机器，但量级稳定）：

| 开销来源 | 相对份额 | 备注 |
|---|---|---|
| Python 对象分配 + 引用计数 + GC | ~35% | 每次运算 3–5 个新对象 |
| 闭包创建与作用域捕获 | ~20% | `_backward` lambda 比普通 Python 函数还贵 |
| 字节码解释（属性查找、动态分派） | ~20% | `.data`、`.grad`、`__mul__` 都要动态查 |
| set/list 维护（`_prev`、`topo`） | ~10% | 反向传播时拓扑排序+集合去重 |
| 失去 SIMD / 向量化 | ~10% | 标量计算无法打包进一条 CPU 指令 |
| 失去 BLAS 级缓存友好访问 | ~5% | numpy 的内存布局 + 分块让 cache 命中率高得多 |

**注意：多线程不在这张表里**——多线程是"能不能把这些开销摊到多个核"的问题，不是"每次开销有多大"的问题。

### 反向传播的额外税

正向慢已经够糟了，反向更贵：
- `topo` 排序遍历所有节点一次（递归 + 集合去重）
- 每个节点调用一次 `_backward()` 闭包
- 闭包里每次 `self.grad += other.data * out.grad` 都是三次属性查找 + 一次加法赋值

几千个节点就是几千次上述开销，全部走 Python 解释器。numpy 的反向是 `dW = dZ @ X.T`——**一次矩阵乘法算完一整层**，和正向共享同一条快车道。

### "单线程" / GIL 的真实角色

GIL 的影响是：**你的单核速度再快也只能用一个核**。所以：

- 如果 micrograd 能用 4 核 → 理论加速 4 倍 → 变成 **25 倍慢**（而不是 100 倍）
- 如果 numpy 只能用 1 核 → 理论减速 4 倍 → 变成 **大约 4 倍慢**（而不是 100 倍快——它依然快，因为 BLAS SIMD 和缓存优化还在）

**两个差距是乘法关系**：100 倍慢 × 4 倍用不上核 = 用户体验上的 400 倍差距。但它们是**两个独立的瓶颈**，必须分别治理。

这也是为什么 Week4 的重构目标是 **numpy 向量化**，不是"给 micrograd 加多进程"——**多进程只能救 4 倍，向量化能救 100 倍**。优先级很清楚。

### 验证：自己测一下

最直接的实验是把两种实现在**同一任务**上对比：

```python
# micrograd 版
t0 = time.time()
for _ in range(10):
    _ = loss_fn(model_micrograd, X, y)
print(f"micrograd 10 forwards: {(time.time()-t0)*1000:.1f} ms")

# numpy 版（等价结构）
t0 = time.time()
for _ in range(10):
    h1 = np.tanh(X @ W1 + b1)
    h2 = np.tanh(h1 @ W2 + b2)
    s  = h2 @ W3 + b3
    loss = ((s - y)**2).mean()
print(f"numpy 10 forwards: {(time.time()-t0)*1000:.1f} ms")
```

**亲眼量一次，比任何估算都有说服力**。两个数字一出来，100 倍这个量级就刻在脑子里了。

### 为什么这很重要（工程哲学）

- **每次运算的常数开销是硬伤**——它不会因为数据多就被摊薄，反而因为运算多就线性放大
- **向量化不是"小优化"，是改变运算的颗粒度**——把 100 万次标量运算打包成 1 次矩阵运算，常数开销被除以 100 万
- **Python 是"胶水语言"的定位**来自这里：慢的是 Python 层，快的是 C 层；深度学习框架的核心就是"让 Python 只在最粗粒度调用 C"
- **初学者容易把慢归因给并发**，因为"多核并行"是个显眼概念。真正的瓶颈往往在更低的层——对象模型、内存布局、SIMD——这些不够显眼但权重更大

### 收获（背下来）

- **100 倍慢的主因是 Python 对象模型和解释器开销，不是单线程**——这两件事属于不同的性能轴
- **每次运算的常数开销**在标量计算里是致命的：1 条 CPU 指令的事情变成几十条字节码，差距是量级的
- **向量化的本质**是"把 N 次小运算合并成 1 次大运算"，常数开销被 N 除——这是 numpy / PyTorch 性能的核心来源，不是"底层用了 C"这么简单
- **GIL 决定 CPU 利用率，不决定单次运算速度**——记住 Q3 和 Q6 的分工：Q6 回答"单次多慢"，Q3 回答"能用几个核"
- **两个瓶颈是乘法关系**：100 倍常数开销 × 4 倍单核独占 = 实际用户感知到的巨大差距；优化时要分别治
- **反向传播的解释器开销 ≥ 正向**：topo 排序 + 每节点一次闭包调用 + 多次属性查找，都在 Python 层反复跑
- **micrograd 慢不是 bug 是定位**：它要把"每次运算都有 _backward 钩子"这件事讲清楚，速度让位于教学清晰度
- **遇到性能问题先问"瓶颈在什么颗粒度"**：标量运算慢就向量化，向量运算慢就换 GPU，I/O 慢就异步——不同颗粒度要不同手段
- **验证性能直觉的唯一方法是 `time.time()` 亲自量**——别凭感觉推，5 行代码就能得到准确量级

## Q7：PyTorch 里 `requires_grad=True` 是干嘛的？启动反向传播的方式有哪些？

**日期**：2026-05-15

### 问题
在 bigram 神经网络法里写：

```python
W = torch.randn((V, V), generator=g, requires_grad=True)
```

`requires_grad=True` 到底标记了什么？除了这种"创建时直接打开"的写法，还有哪些方式让一个张量参与反向传播？以及——光打开开关够了吗，怎么真正"启动"反向？

### 结论先行
- **`requires_grad=True` 只是一个标志位**，告诉 autograd："把以这个张量为叶子的运算记进计算图，将来 `.backward()` 时我要梯度"
- **打开开关 ≠ 启动反向**：开关只是把张量纳入"被求导的集合"，真正触发反向求导的是 `loss.backward()` 或 `torch.autograd.grad(...)`
- **能让张量进入求导集合的方式有 5 种**：① 创建时指定 ② 事后 `requires_grad_()` ③ 用 `nn.Parameter` 包装 ④ 由可求导张量派生（自动传染）⑤ 在 `torch.enable_grad()` 上下文里覆盖局部关闭

一句话压住：**`requires_grad` 是开关，`.backward()` 是按钮**。两件事都做了，梯度才会出现在 `.grad` 里。

### 五种"让张量参与反向传播"的方式

**① 创建时直接指定**（最常见，用于"叶子节点"参数）

```python
W = torch.randn((V, V), generator=g, requires_grad=True)
x = torch.tensor([1.0, 2.0], requires_grad=True)
```

只有**浮点类型**和**复数类型**张量能开 `requires_grad`，整数张量开会报错（梯度对离散量没定义）。

**② 事后用 `.requires_grad_()` 原地修改**

```python
W = torch.randn(V, V)
W.requires_grad_(True)    # 原地打开
W.requires_grad_(False)   # 原地关闭
```

带下划线的方法是 PyTorch 里的"原地操作"约定，**返回 self、不创建新张量**。用于"先加载权重再决定要不要训练"的场景，比如冻结预训练模型某些层。

**③ `nn.Parameter` 包装**（神经网络模块标准写法）

```python
W = nn.Parameter(torch.randn(V, V))   # 默认 requires_grad=True
```

`nn.Parameter` 干了两件事：
- 默认打开 `requires_grad`
- **被自动注册到所在 `nn.Module` 的 `.parameters()` 列表**——优化器靠这个列表更新

只是 `requires_grad=True` 但没包成 `Parameter`，`model.parameters()` 不会收录它，**优化器找不到它**。这是新手最容易踩的坑。

**④ 由可求导张量派生（自动传染）**

只要参与运算的张量里**至少有一个** `requires_grad=True`，输出张量就自动 `requires_grad=True`，并且 `grad_fn` 被填上：

```python
W = torch.randn(V, V, requires_grad=True)
x = torch.randn(V)               # requires_grad=False
y = W @ x + b                    # y.requires_grad == True
                                 # y.grad_fn = <AddBackward0>
```

这就是 §Q5 里 micrograd 那条"原子运算自带 `_backward` → 任意图自动可反向"的 PyTorch 版翻译。**派生出来的中间张量不是叶子**——它的梯度默认不保留，需要调 `.retain_grad()` 才能在 `.grad` 里看到。

**⑤ 在 `torch.enable_grad()` 上下文里覆盖**

```python
with torch.no_grad():
    z = W @ x                # z.requires_grad == False
    with torch.enable_grad():
        y = W @ x            # y.requires_grad == True
```

实际用得少，但理解上下文嵌套的语义对调试有用。

### 真正"启动反向"的方式

打开开关之后，**必须有一个调用去触发反向遍历**：

```python
loss.backward()                          # 最常用：从标量出发反向求导
torch.autograd.grad(loss, [W, b])        # 函数式：返回梯度但不累加到 .grad
loss.backward(gradient=torch.ones_like(loss))  # 非标量损失必须显式给上游梯度
```

两条关键差异：

| | `.backward()` | `torch.autograd.grad()` |
|---|---|---|
| 梯度去哪 | 累加到每个叶子的 `.grad` | 作为返回值，不动 `.grad` |
| 适合场景 | 训练循环（要配 `zero_grad`） | 高阶导、调试、不想污染 `.grad` |

**为什么是"累加"而不是"赋值"？** §Q5 里讲过：同一个参数被多个路径用到时，梯度自然累加是参数共享的正确语义。所以训练循环里**每个 step 开始前必须 `zero_grad`**——否则跨 batch 的梯度会越加越大。

### 反向相关的"关 / 切 / 清"

| 操作 | 作用 | 典型场景 |
|---|---|---|
| `with torch.no_grad():` | 临时关闭整段代码的梯度记录 | 推理、`@torch.no_grad()` 装饰评估函数 |
| `with torch.enable_grad():` | 在 no_grad 内重新打开 | 推理流程里要做对抗攻击之类的求导 |
| `tensor.detach()` | 返回脱离计算图的新张量（共享数据） | 把模型输出当"目标"或喂给另一张图 |
| `tensor.detach_()` | 原地从计算图里断开 | 同上但不想再多分配一个张量 |
| `model.zero_grad()` / `optimizer.zero_grad()` | 把所有参数 `.grad` 置零（或设 `None`） | 每个 training step 开头 |
| `torch.set_grad_enabled(bool)` | 全局开关（不推荐替代上下文管理器） | 一些动态切换训练/评估的代码 |

**`detach()` vs `requires_grad_(False)` 的区别**：前者沿着图剪掉这条边（"我不再追踪上游"），后者只是改当前张量的标志位（"我自己不再贡献梯度"）。两者经常被混用，但语义不同——`detach()` 阻断**上游**梯度回传，`requires_grad_(False)` 阻断**自己**进入求导集合。

### 叶子张量 vs 非叶子张量（最容易踩的坑）

PyTorch 把"用户直接创建的张量"叫**叶子**（leaf），"通过运算派生的张量"叫**非叶子**。

```python
W = torch.randn(V, V, requires_grad=True)   # 叶子
y = W * 2                                    # 非叶子（有 grad_fn）
loss = y.sum()
loss.backward()

print(W.grad)   # 有值
print(y.grad)   # 是 None！警告："非叶子梯度默认不保留"
```

**默认行为**：只有叶子张量的 `.grad` 会被保留。非叶子要想看梯度，调 `y.retain_grad()` 在 `backward()` 之前。这设计是为了省内存——中间梯度在反向传完之后通常就没用了。

**判断是不是叶子**：`tensor.is_leaf`。注意：`W = torch.randn(V, V).cuda().requires_grad_(True)` 里 `.cuda()` 创建了一个**新张量**，原来的 `W` 已经不是这个变量绑的那个，且**新张量是非叶子**（因为 `.cuda()` 是运算）——这种链式写法是 bug 的温床。正确写法是 `W = torch.randn(V, V, device='cuda', requires_grad=True)`。

### 在 bigram 神经网络法里的标准模板

回到题目的代码，完整训练循环长这样：

```python
W = torch.randn((V, V), generator=g, requires_grad=True)

for step in range(epochs):
    # 前向
    xenc = F.one_hot(xs, num_classes=V).float()
    logits = xenc @ W
    counts = logits.exp()
    probs = counts / counts.sum(1, keepdim=True)
    loss = -probs[torch.arange(num), ys].log().mean()

    # 反向
    W.grad = None         # 清零（设 None 比 zero_() 略快，且能让 backward 直接赋值而不是 += 累加到旧 buffer）
    loss.backward()       # 启动反向，填充 W.grad

    # 更新
    W.data += -lr * W.grad   # 用 .data 绕开 autograd，避免这一步被记进图
```

**关键三处细节**：

1. **`W.grad = None` 而不是 `W.grad.zero_()`**：PyTorch 官方推荐前者，因为这样 `backward` 第一次写梯度是赋值，省一次加法；而且如果某个参数本轮没参与计算，它就保持 `None`，不会有"幽灵梯度"
2. **`W.data += ...` 而不是 `W += ...`**：直接对 `W` 做就地加会被 autograd 视为对叶子的修改，触发 `RuntimeError`。通过 `.data` 绕开求导记录是手动训练循环的常见手法；用 `optimizer.step()` 时 PyTorch 内部用 `torch.no_grad()` 做同样的事
3. **`F.one_hot(...).float()` 必须转 float**：one_hot 默认是整数张量，整数不能 requires_grad、也不能参与可求导运算的输入（虽然这里 one_hot 自己不需要梯度，但 `@` 运算需要两边都是浮点）

### 一个反直觉的实验：`requires_grad=True` 加在哪里？

下面两段代码哪段会让 W 拿到梯度？

```python
# A
x = torch.tensor([1., 2.])
W = torch.randn(2, 2, requires_grad=True)
y = (W @ x).sum()
y.backward()
print(W.grad)   # ?

# B
x = torch.tensor([1., 2.], requires_grad=True)
W = torch.randn(2, 2)
y = (W @ x).sum()
y.backward()
print(W.grad)   # ?
```

**A 拿得到**（W 是叶子且开了开关），**B 拿不到**（W 没开开关，autograd 不记录"对 W 怎么变的导数"）。**`requires_grad` 必须打在"你要求梯度的那个张量"上，不是打在"输入"上**。新手常以为"反正运算会传染，随便打一个就行"——但传染只决定**输出是否可求导**，叶子的 `.grad` 只对**自己开了开关的叶子**填充。

### 收获（背下来）

- **`requires_grad` 是开关，`.backward()` 是按钮**——两者都要做，少一个梯度就不会出现在 `.grad` 里
- **5 种打开方式各有用途**：直接创建（参数）、`requires_grad_()`（事后改）、`nn.Parameter`（标准模块）、运算传染（中间张量）、`enable_grad()`（局部覆盖）；**不要忘记 `nn.Parameter` 才会被 `model.parameters()` 收录**——光开 `requires_grad` 优化器找不到它
- **派生张量自动传染 `requires_grad`** 是 §Q5 micrograd 那套"任意图自动可反向"机制在 PyTorch 里的对应——你不需要手动标注图里的每个节点
- **梯度默认累加，不是赋值**——所以每个 step 必须 `zero_grad`（或 `grad = None`，更优）；这条规则的根本理由是参数共享场景下累加才正确
- **`backward()` 写到 `.grad`；`torch.autograd.grad()` 返回梯度不写**——前者用于训练循环，后者用于高阶导和调试
- **`detach()` 剪掉上游边，`requires_grad_(False)` 关掉自己**——两件事容易混，遇到"明明设了不求梯度怎么还在追踪"先想是不是该用 `detach()`
- **非叶子张量的 `.grad` 默认是 `None`**——想看中间梯度调 `retain_grad()`；这是省内存的有意设计，不是 bug
- **手动训练循环要用 `param.data += ...` 而不是 `param += ...`**——后者会触发"叶子被原地修改"的 autograd 错误；`optimizer.step()` 内部用 `torch.no_grad()` 做了等价的事
- **`requires_grad` 必须打在"你要求梯度的张量"上**——传染只管下游"可不可以求导"，不管哪个叶子能拿到 `.grad`
- **整数张量不能 `requires_grad`**——梯度对离散量没定义；one_hot 之类的整型输出在送进可求导运算之前必须 `.float()`
- **`requires_grad=True` 的张量做完 `.cuda()` 或 `.to(device)` 会变成非叶子**——device 放到构造参数里（`torch.randn(..., device='cuda', requires_grad=True)`）才是叶子，这是迁移代码到 GPU 时的常见坑

## Q8：`w.grad = None` 不是把导数释放了吗？那 `backward()` 还怎么反向传播？

**日期**：2026-05-15

### 问题
bigram 训练循环里这样写：

```python
for epoch in range(epochs):
    xenc = F.one_hot(xs, num_classes=V).float()
    logits = xenc @ w
    counts = logits.exp()
    probs = counts / counts.sum(dim=1, keepdim=True)
    nll = -probs[torch.arange(N), ys].log().mean()
    loss = nll + l2 * (w**2).mean()

    w.grad = None         # ← 这一步
    loss.backward()
    with torch.no_grad():
        w -= lr * w.grad
```

直觉上 `w.grad = None` 像是"把导数扔掉"，那紧接着的 `loss.backward()` 凭什么还能算出梯度？是不是把"反向传播要用的东西"也一起释放了？

### 结论先行
**`w.grad` 是"梯度结果缓存"，不是"计算图"**。两件事生命周期完全分开：
- **计算图（DAG）** 在每次前向运算时**动态新建**，挂在中间张量（`logits / counts / probs / loss`）的 `grad_fn` 上，**不在 `w` 身上**
- **`w.grad`** 是上一轮 `backward()` 算完后写到 `w` 的**结果属性**，和图无关
- `w.grad = None` 只清结果缓存，**图一根毛都没碰**；下一句 `loss.backward()` 用的还是这一轮前向刚建好的图

一句话：**清梯度 ≠ 清图**。导数是反向那一刻从图里现算出来的，不是事先存好的。

### 三个独立的对象，必须分清

| 对象 | 存在哪 | 谁创建 | 谁释放 |
|---|---|---|---|
| **`w` 的数据** | `w.data` | `torch.randn(..., requires_grad=True)` | 程序结束 |
| **`w.grad`（梯度结果缓存）** | `w.grad` 属性 | 第一次 `backward()` 写入 | **你手动 `w.grad = None` / `.zero_()`** |
| **计算图（DAG）** | 中间张量的 `grad_fn` | **每次前向时动态构建** | `backward()` 跑完后**自动释放**（默认 `retain_graph=False`） |

**关键**：`w` 是叶子（leaf），叶子的 `grad_fn` 是 `None`——它**不持有图**。图挂在前向产生的中间张量上：

```
xenc ──┐
       ├─→ logits  (grad_fn=MmBackward,    边里持有对 w 的引用)
   w ──┘     │
             ▼
          counts  (grad_fn=ExpBackward)
             │
             ▼
          probs   (grad_fn=DivBackward)
             │
            ...
             ▼
           loss   (grad_fn=MeanBackward)
```

`loss.backward()` 是从 `loss` 出发**反向 DFS 这棵 DAG**，把每条边的局部导数累乘起来填到叶子的 `.grad` 上。整个反向过程依赖的是**图**，不是 `w.grad`。`w.grad` 是反向**输出**写入的目的地，不是反向**输入**读取的来源。

### 一次循环的时间线

```
epoch 0:
  ① 前向：建图 A，挂在 logits/counts/probs/loss 的 grad_fn 上
  ② w.grad = None  ← 此时 w.grad 本来就是 None（首次），是个 no-op
  ③ loss.backward()
     → 沿图 A 反向遍历
     → 把梯度写到 w.grad
     → 自动释放图 A（grad_fn 链被清空）
  ④ with torch.no_grad(): w -= lr * w.grad   # 用 .grad 更新参数

epoch 1:
  ① 前向：建图 B（全新一棵，和图 A 无关）
  ② w.grad = None  ← 这次有用：清掉 epoch 0 留下的梯度缓存
  ③ loss.backward()
     → 沿图 B 反向
     → 写入 w.grad（因为 ② 清过，这一步是赋值不是累加）
     → 释放图 B
  ④ 更新 w.data
```

**图和梯度缓存在时间上是错开的**：

- 图在 ① 这一轮新建，③ 用完后立刻释放
- `w.grad` 在 ③ 写入，留到下一轮 ② 被清

`w.grad = None`（②）卡在 ① 之后、③ 之前——这时图已经建好、梯度还没算，**清梯度不影响后面读图**。

### 为什么必须清？——累加语义

PyTorch 的 `backward()` 是 `w.grad += ...` 而不是 `w.grad = ...`。如果不清：

```
epoch 0: w.grad = g0
epoch 1: w.grad = g0 + g1     ← 错！包含上一轮的旧梯度
epoch 2: w.grad = g0 + g1 + g2
```

累加的设计**不是 bug**，是为了正确处理**同一参数在同一图里被多次使用**的场景（参数共享，见 §Q5 末尾）：

```python
y = w * x1 + w * x2          # w 出现两次
y.backward()                 # w.grad = x1 + x2 ← 累加才正确
```

跨 step 之间的累加没意义，所以训练循环必须主动清。

### `w.grad = None` vs `w.grad.zero_()`

两种写法都能清，但**官方推荐 `= None`**：

| | `w.grad = None` | `w.grad.zero_()` |
|---|---|---|
| 内存 | 释放原 grad 张量，下次 backward 重新分配 | 复用原 buffer，置零 |
| 首次 backward 行为 | **赋值**写入（更快） | 累加到全零（多一次加法） |
| 没参与本轮计算的参数 | 保持 `None`，不出现"幽灵零梯度" | 仍是 0 张量，看起来像参与了但没贡献 |
| `optimizer.step()` 处理 | 跳过 `None` 的参数 | 仍会用 0 梯度更新（不影响结果但浪费） |

老代码里写 `optimizer.zero_grad()`，PyTorch 1.7+ 加了参数 `optimizer.zero_grad(set_to_none=True)`，2.0+ 起这是**默认值**——也就是说现在的 `zero_grad()` 等同于把所有参数的 `.grad` 设为 `None`，和你手写的 `w.grad = None` 是同一件事。

### 如果**真**想"释放反向传播能力"，得改图，不是改 grad

要让 `loss.backward()` 跑不出梯度，有几种合法手段，每一种都是在**动图**而不是动 `w.grad`：

1. **`w.requires_grad_(False)`**：把 `w` 移出"求导集合"，下一轮前向时 `logits = xenc @ w` 就不会建出连到 `w` 的边
2. **`logits = xenc @ w.detach()`**：在这一处切断这条边（其它地方用 `w` 仍可求导）
3. **`with torch.no_grad():`**：把整段前向罩在里面，**根本不建图**
4. **`backward(retain_graph=False)` 后再调一次 `backward()`**：会报 "Trying to backward through the graph a second time"，因为图在第一次 backward 后已自动释放

注意第 4 条正是反向"消耗"图的证据——**`backward()` 默认一图只能用一次**。所以"`backward()` 之后图就没了"和"`w.grad = None`"完全是两条独立的事：

```
释放图     ← backward() 跑完后自动做（或 retain_graph=True 保留）
释放梯度缓存 ← w.grad = None / .zero_() 手动做
```

### 验证：自己打一行就清楚

```python
xenc = F.one_hot(xs, num_classes=V).float()
logits = xenc @ w

print(w.grad_fn)        # None       ← 叶子不持有图
print(logits.grad_fn)   # <MmBackward0> ← 图在这里
print(w.grad)           # None / 上一轮梯度

w.grad = None
print(logits.grad_fn)   # <MmBackward0> ← 完全没变，图毫发无损

loss = ...              # 接着算 loss
loss.backward()         # 照样能反向，w.grad 被重新填上
```

把这段嵌进训练脚本跑一次，"清梯度不动图"这件事就刻进印象了——比读十遍文档管用。

### 常见误解纠正

> **误解 1**："`w.grad = None` 会让 `backward()` 没东西可算。"
>
> 错。`backward()` 的输入是**图**（在中间张量上）和**起点张量**（`loss`），输出才是 `w.grad`。清掉**输出**不影响读取**输入**。

> **误解 2**："图存在 `w` 上，所以动 `w` 就动了图。"
>
> 错。叶子的 `grad_fn` 永远是 `None`。图是非叶子张量的属性。

> **误解 3**："如果 `w.grad` 不清，下一次 `backward()` 就不工作了。"
>
> 错。不清也能工作，只是结果是**累加**——这是 bug 的来源（每个 step 的更新方向被历史污染），不是报错的来源。

> **误解 4**："`backward()` 之后再次 `backward()` 会读旧的 `w.grad`。"
>
> 错。第二次 `backward()` 报的错是"图已被释放"，和 `w.grad` 状态无关。要二次反向得加 `retain_graph=True`。

### 收获（背下来）

- **梯度缓存 ≠ 计算图**：`w.grad` 是 `backward()` 的**输出目的地**，图是 `backward()` 的**输入来源**。清前者不影响读后者
- **图挂在非叶子张量的 `grad_fn` 上，不在叶子参数上**——叶子的 `grad_fn` 永远是 `None`，所以参数本身从不"持有图"
- **图在每次前向时动态新建，在 `backward()` 后自动释放**（默认 `retain_graph=False`）——这是 PyTorch "define-by-run" 的核心特征，和静态图框架（旧 TF 1.x）不同
- **梯度累加是设计，不是 bug**：同一参数在图里被多次使用时累加才正确（参数共享）；跨 step 累加没意义，所以训练循环必须主动清
- **`w.grad = None` 优于 `w.grad.zero_()`**：首次 backward 是赋值省一次加法；没参与计算的参数保持 `None` 不会有"幽灵零梯度"；PyTorch 2.0+ 的 `optimizer.zero_grad()` 默认就是这种行为
- **想"关掉反向"要动图，不要动 grad**：`requires_grad_(False)` / `detach()` / `torch.no_grad()` 三种手段对应三种粒度（整张图、单条边、整段代码）
- **遇到"梯度怎么是上次的两倍"**先怀疑没清 grad；遇到"再次 backward 报错"是图被释放，要 `retain_graph=True`——两种症状对应两件不同的事
- **理解 PyTorch 的关键是把"数据 / 图 / 梯度"三件事分开看**：动态图框架最容易混淆的就是它们的生命周期；想清楚每个对象"谁建、谁用、谁释放"，疑难问题就自动消失了

## Q9：为什么 bigram 神经网络法 lr=50 能稳，同样的 lr 放到 MLP 上立刻发散？

**日期**：2026-05-15

### 问题
`nn_based.py` 里训练 bigram 用了 `lr=50`，nll 平稳下降到 2.45 附近。但 §Q4 里 MLP 配 `lr=1.0` 就直接 `exp` 溢出。同一个超参为什么差 50–500 倍？是 bigram "命好"还是有原理？

### 结论先行
**bigram 不是命好，是问题简单到同时拿到三张"稳定性优惠券"，MLP 一张都拿不到**：

1. **损失全局凸**——softmax + cross-entropy + 单层线性 = 多类逻辑回归，是凸优化教科书例子
2. **梯度天然有界**——softmax + CE 对 logits 的梯度恒满足 $\|\nabla\|_2 \le \sqrt{2}$，与参数大小无关
3. **one-hot 输入 $\|x\|^2 = 1$**——相当于免费送了输入归一化，把 §Q4 的 $\eta_{crit} \approx N/\|h\|^2$ 推到 $O(N)$ 量级

把这三件事乘起来，bigram 的安全 lr 区间从 MLP 的 $O(0.01)$ 直接跳到 $O(N) \sim O(10^5)$——**七个数量级的差距**。lr=50 看似激进，对应到 §Q4 的 $\alpha = \eta \cdot \|\nabla^2 L\|_\text{eff}$ 上其实仍然 $\alpha < 1$，温和得很。

一句话压住：**lr 大不大不是看数字，看 $\alpha = \eta \cdot$ 局部曲率。曲率被压到地板时，$\eta$ 自然能放飞**。

### 复习：发散判据（沿用 §Q4 的框架）

一维线性化下，参数更新对损失的影响倍数：

$$
\alpha = \eta \cdot \|\nabla^2 L\|_\text{eff} \quad (\alpha < 2 \text{ 才不发散})
$$

所以"安全 lr"≈ 1 / 局部曲率。bigram 把曲率压到极小，MLP 把曲率拉到极大——同样的 lr 落到两边的 $\alpha$ 上完全不是一个量级。

### 优惠券 1：损失全局凸

bigram 的损失函数：

```python
loss = cross_entropy(softmax(xenc @ W), y)
```

是经典的**多类逻辑回归**——凸优化教材的标准例子，$L(W)$ **全局只有一个最小值**，没有局部最小、没有鞍点、没有恶性曲率方向。

凸地形的红利：

- 步子迈大了顶多**震荡**（跨过最小值再跨回来），不会"飞出去再也回不来"
- 即使 lr 大到震荡，只要 $\alpha < 2$ 就还在收敛区间，仍然向下走
- 没有"被推到一个曲率超大的烂地方"这种灾难

**MLP 失去这张券**：ReLU/tanh + 多层 = 严重非凸。地形里有刀峰、有平原、有悬崖。lr=50 在凸碗里只是震荡，在 MLP 地形里就是 §Q4 里那个 `exp` 溢出——一脚踏进悬崖。

### 优惠券 2：softmax + cross-entropy 的梯度自带上界

数学上最干净的一条。对 logits 求导：

$$
\frac{\partial L}{\partial \text{logits}_i} = p_i - \mathbb{1}[i = y]
$$

其中 $p_i = \text{softmax}(\text{logits})_i \in [0, 1]$。所以对每个样本：

$$
\left\|\frac{\partial L}{\partial \text{logits}}\right\|_2 \le \sqrt{2}
$$

**永远！不管 logits 多大、不管 W 跑到哪。** 因为 $p_i$ 被 softmax 压在 $[0,1]$，残差 $p_i - \mathbb{1}[i=y]$ 自动被压在 $[-1, 1]$。

对比 §Q4 的 MSE：梯度 $\propto (s - y)$，**没有上界**——参数推得越偏，梯度越大，正反馈到爆炸。这是"MSE + 线性输出"配 lr=1 都炸、"softmax + CE + 线性输出"配 lr=50 没事的根本原因。

> **记死**：softmax + CE 是"自带刹车"的损失；MSE 是"油门越深越想加油"的损失。

**反向验证**（来自 §Q4）：Karpathy micrograd demo 用 hinge 也能 lr=1.0 不炸，因为 hinge 在分类正确时梯度精确为 0——同样是"损失自带刹车"机制。**有界梯度损失 = 优惠券 2 的通用形式**。

### 优惠券 3：one-hot 输入让梯度尺度归一化

bigram 的输入是 one-hot：

$$
\|x_\text{enc}\|_2^2 = 1 \quad\text{（只有一个位置是 1）}
$$

代回 §Q4 的发散公式 $\eta_{crit} \approx N / \|h\|^2$：

| 输入类型 | $\|x\|^2$ 量级 | 安全 $\eta$ 量级 |
|---|---|---|
| **one-hot（bigram）** | **1** | **N**（很大！） |
| 归一化的 dense 向量 | ~ 维度 (e.g. 784) | N / 784 |
| 未归一化的图像像素 | ~ $10^5$ | N / $10^5$ |

bigram 用 one-hot 等于**免费送了一个输入归一化**——这是新手最容易忽略的"隐形优势"。

**额外的稀疏性礼物**：`xenc @ W` 在数学上等价于"取 W 的某一行"。每个样本**只更新 W 的一行**，行与行之间梯度互不干扰。等效维度从 27×27=729 降到了 27——**参数耦合极小**，进一步压低有效曲率。

### 三张券叠加的效果（一张表看清差距）

| | bigram (`nn_based.py`) | 典型 MLP (§Q4) |
|---|---|---|
| 损失曲率上界 | 凸 + 残差 ∈ [-1,1] + ‖x‖=1 → 极小 | 非凸 + 多层放大 → 大且变化剧烈 |
| 梯度上界 | $\sqrt{2}$，与 W 无关 | 无上界，正比于 $(s-y)$ |
| 步长安全区 $\alpha < 2$ | $\eta < O(N)$ | $\eta < O(N / \text{层数} / \text{宽度})$ |
| 实测安全 lr | **10–100** 都能用 | 0.01–0.1 量级 |

bigram 的 lr=50 看似离谱，**对应到 $\alpha$ 上其实很温和**——曲率小到地板，乘起来 $\alpha$ 仍 < 1。

### MLP 为什么一张券都拿不到

1. **多层非线性 → 非凸**：失去优惠券 1
2. **回归任务用 MSE / 输出层无激活 + 大输出值**：失去优惠券 2（分类用 CE 时还能保留这条）
3. **隐层输入是 dense 激活**：$\|h\|^2$ 跟宽度同阶，失去优惠券 3
4. **额外负担**：梯度反向时被 $W^L \cdot W^{L-1} \cdots$ 连乘，深度本身就是放大器（§Q5 末尾链式法则）

把这四件事乘起来，安全 lr 从 $O(N) \approx 10^5$ 直接掉到 $O(0.01)$——和实测吻合。

### 一个常被忽略的推论：lr 调参经验不可跨场景搬运

读教程时最常见的陷阱：

> "Karpathy 的 bigram 用了 lr=50，那我的 MLP 也试试。"

**这个推理在数学上完全错误**，因为 lr 的安全上界由"损失结构 + 网络结构 + 输入尺度"三件事联合决定，bigram 拿到的三张券放到 MLP 上一张都没了。教程里的 lr 值是该具体配置下的经验值，**不是普遍真理**。

更广义的版本：**任何超参的"建议值"都是配套环境的产物，不是常数**。batch size、weight decay、optimizer 选择都遵循同一逻辑。

### 怎么估自己代码的安全 lr 上界

不用解析，**用一次实验估**（推荐 lr-range test）：

1. 选一个非常小的 lr（如 1e-6）
2. 每个 step 把 lr 乘以一个固定倍数（如 1.1）
3. 同时画 loss vs lr
4. 观察曲线：先平、后下降、最后陡升发散
5. **取"陡升前 1/10"**作为安全 lr

bigram 上跑这个 test，loss 在 lr ≈ 100 处开始炸，"安全选 10"——和你 `nn_based.py` 的 lr=50 在量级上一致。MLP 上跑同样的 test，发散点会在 lr ≈ 0.5 附近——配 0.05 才稳。这个方法不用懂任何数学，5 分钟出结果，比死记教程的 lr 数字可靠得多。

### 收获（背下来）

- **lr 大不大看 $\alpha = \eta \cdot$ 局部曲率，不看数字**——曲率被压到地板时 lr 自然能飞，曲率高耸时 lr 必须缩
- **bigram 同时拿三张稳定性券**：损失全局凸 + softmax-CE 梯度有界 + one-hot 输入归一化；三张券叠起来把安全 lr 推到 $O(N)$
- **softmax + cross-entropy 的梯度恒 $\le \sqrt{2}$**——这是它配大 lr 不炸的根本原因，也是分类任务首选 CE 而不是 MSE 的硬理由
- **one-hot 是隐形输入归一化**：$\|x\|^2 = 1$ 直接把 §Q4 的 $\eta_{crit} \approx N/\|h\|^2$ 推到 $O(N)$；用 dense 输入时这条优惠就消失了
- **稀疏输入还有"参数解耦"的二次红利**：one-hot @ W 只激活 W 的一行，跨行梯度互不干扰，进一步压低有效曲率
- **MLP 失去全部三张券**：非凸地形 + 多层梯度连乘 + dense 隐层激活，把安全 lr 直接打到 $O(0.01)$ 量级
- **教程里的 lr 数字不是常数**：是该配置下的经验值；换损失、换网络、换输入尺度都得重估
- **估安全 lr 用 lr-range test，不要凭感觉**：5 分钟跑一次比读十遍论文的 lr 表都准
- **§Q4 / §Q5 / §Q9 的因果链是一致的**：损失结构决定梯度上界，网络结构决定曲率放大倍数，两者共同决定 lr 安全区间——理解这条链条比记任何具体数字都有用

## Q10：hinge (SVM max-margin) 和 MSE 到底差在哪？为什么分类任务不能用 MSE？

**日期**：2026-05-15

### 问题
Karpathy EP1 的 make_moons demo 用的是 **hinge loss**，而我在 §Q4 用 MSE 配 lr=1.0 直接炸掉。两者形式上都是"算预测和目标的差"，但行为差得离谱：
- MSE 把分类任务搞发散
- hinge 在同样的 lr 下稳如老狗

这是巧合还是必然？为什么"分类用 MSE 不行"在工程上是常识，但很少有人讲清楚**到底哪里不行**？

### 结论先行

**hinge 和 MSE 的差异不是"两个公式不同"，是两套完全不同的损失哲学**：

- **MSE**："预测精确等于目标"——目标是 $\hat y = y$，**双侧惩罚**任何偏离
- **hinge**："预测站在正确一侧且有足够 margin"——目标是 $y \cdot \hat y \ge 1$，**单侧惩罚**，过了 margin 之后**完全免费**

这两种哲学决定了它们的**梯度结构**完全不同，进而决定了：
1. **稳定性**（§Q4、§Q9 的优惠券 2 直接来自这里）
2. **优化行为**（hinge 自动聚焦于"难样本"，MSE 无差别推所有样本）
3. **适用场景**（MSE → 回归，hinge → 分类决策边界，CE → 概率分类）

一句话压住：**loss 的选择本质是"对预测错误的处罚函数"的选择——这个函数的形状决定了优化器看到的地形**。形状选错，再好的 lr / 初始化也救不回来。

### 数学定义对比

**MSE（Mean Squared Error）**：

$$
L_\text{MSE} = \frac{1}{N} \sum_i (\hat y_i - y_i)^2
$$

**Hinge loss**（二分类，$y \in \{-1, +1\}$，$\hat y \in \mathbb{R}$）：

$$
L_\text{hinge} = \frac{1}{N} \sum_i \max(0,\; 1 - y_i \cdot \hat y_i)
$$

把 $y_i \cdot \hat y_i$ 这个乘积叫做 **margin**：
- margin > 0：预测方向对（分类正确）
- margin > 1：预测方向对**且有充足把握**（hinge 不再罚）
- margin < 0：预测方向错（分类错误）

### 形状对比图（$y = +1$ 时，横轴 $\hat y$）

```
loss
 ↑                  MSE
 |     \           /
 |      \         /
 |       \       /        ← MSE 是抛物线，双侧惩罚
 |        \     /            最低点在 ŷ=1，"过度正确"也罚
 |         \   /
 |          \./
 |___________|___________ ŷ
            1

loss
 ↑\
 | \
 |  \                     ← Hinge 是分段线性
 |   \                       margin < 1 时线性下降
 |    \                      margin ≥ 1 时死平为 0
 |     \________________
 |______________________ ŷ
        1
```

**关键观察**：
- MSE 在 $\hat y = 1$ 处导数为 0，**两侧都向中心拉**——预测 +5 时它会把你拉回 +1，这对**回归**是对的，对**分类**是灾难
- hinge 在 $\hat y \ge 1$ 处导数恒为 0，**只罚错误那一侧**——分对了就放过你，不再消耗优化资源

### 梯度结构对比（这部分是核心）

**MSE 对 $\hat y$ 的梯度**：

$$
\frac{\partial L_\text{MSE}}{\partial \hat y_i} = \frac{2}{N}(\hat y_i - y_i)
$$

**特性**：无上界。预测越偏离 $y$，梯度幅度越大。这是 §Q4 那个发散回路的源头：

$$
\underbrace{\hat y \text{ 偏离 } y}_\text{预测错} \;\Rightarrow\; \underbrace{|\nabla L| \text{ 变大}}_\text{梯度大} \;\Rightarrow\; \underbrace{|\Delta w| \text{ 变大}}_\text{步长大} \;\Rightarrow\; \underbrace{\hat y \text{ 推得更偏}}_\text{预测更错} \;\Rightarrow\; \cdots
$$

正反馈环，越走越远。

**Hinge 对 $\hat y$ 的梯度**：

$$
\frac{\partial L_\text{hinge}}{\partial \hat y_i} =
\begin{cases}
-y_i / N & \text{当 } y_i \hat y_i < 1 \text{（分错或没够 margin）} \\
0 & \text{当 } y_i \hat y_i \ge 1 \text{（分对且够自信）}
\end{cases}
$$

**特性**：
1. **恒有上界**：$|\nabla L| \le 1/N$，**与预测幅度无关**——这就是 §Q9 优惠券 2 的另一个版本
2. **分对样本梯度精确为 0**：不浪费一点优化资源在已经搞定的样本上
3. **断点处不可导**（$y \hat y = 1$ 那一点），但实践里用次梯度即可，没问题

> **这两个梯度结构决定了一切**：MSE 是"油门越深越想加油"，hinge 是"过了线就熄火"。§Q4 那个发散回路在 hinge 下根本搭不起来，因为梯度有界 → 步长有界 → 没有正反馈。

### 把两者放进 §Q4 的发散判据

回到 §Q4 推的发散公式：

$$
\alpha = \eta \cdot \frac{2}{N} \|h\|^2 \cdot \text{(梯度对参数的二阶项)}
$$

**MSE**：二阶项 ∝ 1（恒定），所以 $\alpha$ 完全由 $\eta \cdot \|h\|^2 / N$ 决定。$\|h\|^2$ 在训练中会**随参数膨胀而膨胀**，所以一旦失控，$\alpha$ 也跟着指数膨胀——发散一发不可收拾。

**Hinge**：二阶项要分段算。对于"已经够 margin"的样本（训练后期占绝大多数），二阶项 = 0，$\alpha = 0$——这部分样本**完全不参与稳定性分析**。只有少数"支持向量"贡献 $\alpha$，而它们的梯度还有 $1/N$ 的上界。**$\alpha$ 被天然压在很小的值，无论 lr 多大**。

这就是为什么 Karpathy 的 demo 配 lr=1.0 能稳——hinge 把发散公式里**所有的放大项都堵住了**：

| 放大项 | MSE | Hinge |
|---|---|---|
| 梯度幅度 | 无上界，随预测偏差线性增长 | 上界 $1/N$，与预测无关 |
| 参与样本数 | 全部 | 只有"难样本"（违反 margin 的） |
| 训练后期行为 | 仍在全力推所有样本 | 大部分样本梯度 = 0，自动停训 |

### 适用场景的根本原因

为什么 MSE 是回归默认而 hinge 是分类工具？**不是约定俗成，是数学结构使然**：

**回归任务**：
- 真实值 $y \in \mathbb{R}$ 在连续尺度上有意义
- "预测 100，真实 102" 和 "预测 100，真实 103" 的区别**是有意义的**
- 高斯噪声假设下，MSE 是最大似然估计（数学上有依据）
- 双侧惩罚合理：预测过高过低都要罚

**分类任务**：
- 真实标签 $y \in \{-1, +1\}$（或 one-hot）是**离散的**
- "预测 +5，真实 +1" 比 "预测 +1.01，真实 +1" **没有任何劣势**——两者都分类正确
- MSE 会把 +5 拉回 +1，**这是错误的优化目标**——你逼模型"精确等于 +1"是在过度拟合到一个数值，而不是在学决策边界
- hinge / CE 才反映分类任务的真正目标："分对就行，越自信越好"

**一个极端例子**：训练集里有个样本 $\hat y = +100, y = +1$，从分类角度这是"非常自信的正确"——MSE 给它打 $(100-1)^2 = 9801$ 的惩罚，**会主导整个 batch 的梯度方向，把模型拉得乱七八糟**。hinge 给它打 0 分，完全忽略。

### 收获（背下来）

- **loss 选择 = "对错误的惩罚函数"选择**：函数形状决定优化地形，地形决定能用多大 lr、训练后期是否稳定、能否收敛到合理解
- **MSE 是"对称双侧惩罚 + 平方放大"**：适合回归（连续目标）；用于分类时会过度惩罚"过度自信的正确"，并且失去 §Q9 的优惠券 2（梯度无上界 → 配大 lr 发散）
- **hinge 是"单侧惩罚 + 过 margin 后免费"**：天然实现"分对样本零梯度"，自动聚焦难样本，这是 SVM "支持向量" 概念的源头
- **梯度有上界 = 配大 lr 安全**：hinge 的 $|\nabla| \le 1/N$ 直接堵死了 §Q4 的发散回路；这也是 §Q9 bigram 配 lr=50 不炸的同款机制（softmax+CE 梯度有界 $\le \sqrt{2}$）
- **训练后期行为完全不同**：MSE 还在推所有样本，hinge 只在推"违反 margin 的少数样本"——这就是为什么 hinge 训练 loss 真的能干净降到 0，MSE 几乎永远到不了
- **断点不可导不是问题**：hinge 在 $y\hat y = 1$ 处不可导，但次梯度（取左导数或右导数都行）在工程实现里完全够用——`max(0, 1-yŷ)` 直接用 `(1 - y*ŷ).relu()` 一行搞定
- **`(1 - y*ŷ).relu()` 是 hinge 的标准实现**：复用 ReLU 算子，这就是 Karpathy 给 micrograd 加 ReLU 的另一个理由——既是激活，也是 hinge 的零件
- **现代 DL 分类几乎一律用交叉熵不是 hinge**：CE 的概率视角更自然、梯度更平滑、配 softmax 推导极简（§Q9 推过的 $\partial L/\partial \text{logits} = p - y$）；hinge 主要在 SVM、对比学习 (contrastive)、triplet loss、对抗训练里出现
- **margin 思想超越 SVM**：所有"分对还不够，要分得有距离"的损失都是 margin 家族（triplet、ArcFace、对抗 margin）；hinge 是这家族的入门款，理解它就理解了 metric learning 的根
- **回归任务遇到离群点别死用 MSE**：MSE 对离群点过敏（平方放大），换 Huber / L1 / 分位数损失能稳很多；这是"loss 形状 → 鲁棒性"的另一个例证
- **教程里"分类用 MSE 也能跑"是错觉**：能跑不等于跑得对——MSE 在分类上能收敛是因为问题简单，问题一难（多类、不平衡、深网）立刻翻车；从一开始就用 CE / hinge 是省事的做法
- **§Q4 / §Q9 / §Q10 是同一条因果链的三个切面**：§Q4 揭示"梯度无上界 → 配大 lr 发散"，§Q9 揭示"梯度有上界 → 配大 lr 安全"，§Q10 把上界这件事直接归到**损失函数的形状**——这三道题串起来就是"为什么神经网络能/不能用大 lr"的完整答案
- **下次选 loss 时先问三个问题**：① 任务是回归还是分类？② 梯度有没有自然上界？③ 训练后期已经做对的样本会不会继续干扰优化？——这三个问题答完，loss 选择基本就定了

## Q11：交叉熵（Cross-Entropy）到底是什么？为什么是分类任务的工业标准？

**日期**：2026-05-15

### 问题
§Q10 把 MSE 和 hinge 讲清楚之后，剩下一个分类任务里出场率最高的损失没讲——**交叉熵**（cross-entropy, CE）。在 §Q9 推 bigram 时它是默认选择，§Q5 第 2 周手推 `dZ2 = A2 - Y` 也是它的产物，但都没正面讲过：
- CE 的公式是怎么来的？为什么是这个形式不是别的？
- 为什么它配 softmax 是"工业标准"？是约定俗成还是有数学硬理由？
- 它和 hinge 都是分类 loss，到底差在哪？

### 结论先行

**CE 是"用错误分布编码真实分布所需的额外信息量"——一个从信息论 / 最大似然 / 编码理论三个独立角度都能推出同一公式的损失函数**。它统治分类任务不是巧合：

1. **数学自然**——三个独立视角（统计、信息论、编码）通向同一公式，证明它"在数学上是分类任务的正确答案"
2. **梯度完美化简**——softmax + CE 配对时梯度恰好是 $\hat p - y$，所有复杂雅可比项消干净，§Q9 优惠券 2（梯度上界 $\sqrt{2}$）和 §第 2 周 `dZ2 = A2 - Y` 的根本来源都在这里
3. **自信犯错惩罚无穷**——$-\log \hat p_y$ 在 $\hat p_y \to 0$ 时趋向 $+\infty$，迫使模型立刻修正"自信地犯错"
4. **永远不归零**——和 hinge 不同，分对样本的梯度仍然 > 0，模型一直被推着"再自信一点"，决策边界更稳健

一句话压住：**CE 在数学上是 softmax 的天然伴侣，配对后所有数学复杂性消失；它是 §Q4/§Q9/§Q10 那条"梯度上界 → 训练稳定"因果链在分类任务上的最终答案**。

### 数学定义

**二分类（BCE）**，标签 $y \in \{0, 1\}$，预测 $\hat p \in (0, 1)$：

$$
L_\text{BCE} = -\big[\, y \log \hat p + (1-y) \log(1-\hat p) \,\big]
$$

**多分类（CCE）**，标签 $y$ 是 one-hot 向量，预测 $\hat p$ 是 softmax 输出：

$$
L_\text{CE} = -\sum_{k=1}^{K} y_k \log \hat p_k
$$

因为 $y$ 是 one-hot，求和里只有正确类那一项非零，化简为：

$$
\boxed{L_\text{CE} = -\log \hat p_{y_\text{true}}}
$$

**最干净的形式——只看模型给"正确类"分配了多少概率，取负对数**。这就是 §Q9 bigram 那行：

```python
loss = -probs[torch.arange(N), ys].log().mean()
```

### 三视角推导（CE 公式的"数学硬证据"）

CE 不是拍脑袋写的，三个完全独立的角度都能推出同一公式。**任意一个视角懂了就够，但能看到三个视角通向同一答案，是 CE 在数学上"对"的最强证据**。

**视角 1：最大似然估计（统计视角）**

模型给样本 $i$ 的真实类分配的概率是 $\hat p_{y_i}$。所有样本独立，联合似然：

$$
\mathcal{L}(\theta) = \prod_i \hat p_{y_i}(\theta)
$$

最大化似然 = 最大化对数似然 = 最小化**负**对数似然：

$$
-\log \mathcal{L} = -\sum_i \log \hat p_{y_i}
$$

**这就是 CE。** 所以 PyTorch 里 `nn.CrossEntropyLoss` 和 `nn.NLLLoss`（Negative Log Likelihood）本质同义——区别只在前者内置 softmax，后者要你自己做 log。

**视角 2：信息论（KL 散度）**

**信息熵**：分布 $p$ 的不确定性 $H(p) = -\sum p_k \log p_k$

**KL 散度**：用 $\hat p$ 近似 $p$ 的"信息损失"：

$$
D_\text{KL}(p \| \hat p) = \sum p_k \log \frac{p_k}{\hat p_k} = \underbrace{-\sum p_k \log \hat p_k}_\text{交叉熵 } H(p, \hat p) \;-\; \underbrace{\big(-\sum p_k \log p_k\big)}_\text{真实熵 } H(p)
$$

整理：

$$
H(p, \hat p) = H(p) + D_\text{KL}(p \| \hat p)
$$

训练时 $p$（真实标签）固定，$H(p)$ 是常数——**最小化交叉熵 = 最小化 KL 散度 = 让预测分布尽量逼近真实分布**。

**视角 3：编码理论（最直观）**

Shannon 编码定理：用最优编码方案表达来自分布 $p$ 的符号，平均每符号需要 $H(p)$ bit。

如果你**用 $\hat p$ 的编码方案去编码来自 $p$ 的符号**（即误判了真实分布），平均每符号需要 $H(p, \hat p)$ bit。**多出来的 bit 就是 KL 散度，就是你的"模型错误"在信息论上的代价**。

> 三个视角通向同一公式——这是 CE "数学上自然"的硬证据。MSE 和 hinge 都没有这种多视角统一性，它们是有用的工具，但不像 CE 那样"几何上、概率上、信息上同时是对的"。

### 梯度结构：CE + softmax 的封神时刻

**这是 CE 在工程上统治分类任务的核心原因**。配合 softmax 时，对 logits $z$ 的梯度有一个魔法般的形式：

$$
\boxed{\frac{\partial L_\text{CE}}{\partial z_i} = \hat p_i - y_i}
$$

**就是预测概率减真实标签**。没有任何复杂项。

**自己手推一遍（值得做一次）**：

设 $\hat p_i = \frac{e^{z_i}}{\sum_k e^{z_k}}$，one-hot 标签下：

$$
L = -\log \hat p_{y_\text{true}} = -\log \frac{e^{z_{y_\text{true}}}}{\sum_k e^{z_k}} = -z_{y_\text{true}} + \log \sum_k e^{z_k}
$$

对 $z_i$ 求导：

$$
\frac{\partial L}{\partial z_i} = -\mathbb{1}[i = y_\text{true}] + \frac{e^{z_i}}{\sum_k e^{z_k}} = \hat p_i - \mathbb{1}[i = y_\text{true}]
$$

**关键点**：softmax 的雅可比是 $K \times K$ 的恶心矩阵（$\partial \hat p_i / \partial z_j = \hat p_i(\delta_{ij} - \hat p_j)$），但 cross-entropy 的对数求导**正好把这个雅可比的复杂部分全消掉**，只剩一个减法。这是数学上的奇迹——其它任何 loss + softmax 的组合都没有这么干净的梯度。

**和 §第 2 周 `dZ2 = A2 - Y` 完美对上**：当年你以为是"巧合化简"，现在能从最大似然 / KL / 信息论三个角度独立推出同一答案——**不是巧合，是 softmax + CE 在数学上配对完美**。

### 三种 loss 的梯度对比（缝合 §Q4 / §Q9 / §Q10）

| 维度 | CE + softmax | MSE | hinge |
|---|---|---|---|
| 梯度公式（对输出）| $\hat p - y$ | $2(\hat y - y)$ | $-y$ 或 $0$ |
| 梯度上界 | $\sqrt{2}$（§Q9 已证） | **无上界**（§Q4 发散源头）| $1$（§Q10 已证）|
| 训练后期分对样本梯度 | 接近 0 但**永不为 0** | 接近 0 | **精确等于 0** |
| 对"自信地犯错"的惩罚 | $-\log \hat p_y \to +\infty$ | 平方增长 | 线性增长 |
| 输出有概率意义？ | ✓（softmax）| ✗ | ✗ |
| 多分类天然支持？ | ✓ | 勉强 | 需要 multiclass hinge |

**§Q9 那张"三张优惠券表"里的优惠券 2（梯度上界 $\sqrt{2}$）就是从 CE + softmax 这里来的**。bigram 配 lr=50 不炸，根本上是 CE 的梯度结构在兜底。

### CE 对"自信犯错"的惩罚（这是它训练效率高的核心）

| 预测正确类的概率 $\hat p_y$ | CE loss | MSE loss（设 $\hat y = 1 - \hat p_y$）|
|---|---|---|
| 0.99 | 0.01 | 0.0001 |
| 0.5 | 0.69 | 0.25 |
| 0.1 | 2.30 | 0.81 |
| 0.001 | **6.91** | 0.998 |
| $10^{-10}$ | **23.0** | ≈1 |

**CE 对"自信地犯错"的惩罚是对数级、无界的**——预测 $\hat p_y = 10^{-10}$ 时 loss = 23。MSE 在同样错误程度下 loss 接近 1，完全没意识到这是个灾难性错误。

**工程含义**：CE 训练时，最自信地犯错的样本会**主导梯度方向**，模型被迫立刻修正它们。这就是 CE 比 MSE 在分类上收敛快得多的根本原因——**它把优化资源精确分配给了最该修的样本**。

### 与 hinge 的对比：两种分类哲学

| 维度 | Cross-Entropy | Hinge |
|---|---|---|
| 哲学源头 | 概率 / 信息论 | 几何 / margin |
| 模型输出 | 概率分布（softmax 后）| 实数分数（任意）|
| 关心什么 | 整个概率分布的形状 | 决策边界两侧的间隔 |
| 训练后期分对样本 | 仍有小梯度（$\hat p_y < 1$ 永远成立）| **精确零梯度** |
| 能给"置信度"吗 | ✓（直接读 $\hat p$）| ✗（分数无概率意义）|
| 多分类扩展 | 天然（softmax + 多项 CE）| 需要 multiclass hinge |
| 在 NN 里的地位 | **工业标准** | 小众，主要在 SVM / 对比学习 |

**关键差异**：CE **永远没法**把分对样本梯度精确推到 0。只要 $\hat p_y < 1$，$-\log \hat p_y > 0$，就还有惩罚。

- **好处**：模型一直被鼓励"再自信一点"，决策边界更稳健，泛化更好
- **坏处**：训练永远到不了 loss = 0（除非过拟合），不像 hinge 能干净归零；这意味着**永远存在过拟合风险**——L2 正则、early stopping、dropout 之类的手段就是来对冲它的

### 数值稳定性：log-sum-exp trick

**朴素实现**（绝对不要这么写）：

```python
probs = softmax(logits)
loss = -log(probs[y_true]).mean()
```

两个问题：
1. softmax 里 $e^{z_k}$ 在 $z_k > 709$ 时 float64 溢出（float32 是 88）
2. 即使没溢出，$\hat p_y$ 可能小到 $10^{-100}$，log 之后变 $-\infty$

**正确实现**（log-sum-exp）：

$$
\log \hat p_i = z_i - \log \sum_k e^{z_k} = z_i - \max(z) - \log \sum_k e^{z_k - \max(z)}
$$

减去 $\max(z)$ 让 exp 里最大项变成 $e^0 = 1$，其它项更小——**绝对不会溢出**。PyTorch 的 `F.cross_entropy(logits, targets)` 和 `F.log_softmax` 内部都这么实现。

**接口陷阱**：

```python
# ❌ 错误：会有数值问题
loss = -torch.log(F.softmax(logits, dim=1))[range(N), y].mean()

# ❌ 双重错误：F.cross_entropy 内部已做 log_softmax，传 softmax 后的概率会重复计算
loss = F.cross_entropy(F.softmax(logits, dim=1), y)

# ✓ 正确：传 logits，让框架内部处理稳定性
loss = F.cross_entropy(logits, y)
```

**`nn.CrossEntropyLoss` 吃 logits 不吃概率**——这是新手最容易踩的接口坑，且不易察觉（数值不会报错，只是结果错）。

### 接回 §Q4 → §Q9 → §Q10 的因果链：四题闭环

| Q | 揭示的因果 |
|---|---|
| §Q4 | 梯度无上界（MSE 配线性输出） → 配大 lr 发散 |
| §Q9 | 梯度有上界（softmax+CE，bigram） → 配大 lr 安全 |
| §Q10 | hinge 的梯度有上界且分对样本零梯度 → 配大 lr 安全 |
| **§Q11** | **softmax + CE 是数学上最自然的分类损失**：从最大似然 / KL / 编码三个角度都推出同一形式；梯度化简成 $\hat p - y$，是 §Q9 优惠券 2 的根本来源；§第 2 周 `dZ2 = A2 - Y` 也是它的产物 |

**完整论点**：loss 的梯度结构决定优化稳定性。CE 因为**概率本质 + 完美的梯度化简 + 对自信错误的对数惩罚 + 自然的数值稳定**，成为分类任务的最优解。

### CE 的现代变种和家族

CE 不是一个孤立的 loss，是一个**家族的总司令**。现代 DL 里 90% 的"看起来不是 CE"的损失，本质都是 CE 的伪装：

| 变种 | 用途 | 与 CE 的关系 |
|---|---|---|
| **Binary CE (BCE)** | 二分类、多标签分类 | CE 在 2 类时的退化 |
| **NLL Loss** | 自定义概率模型 | 和 CE 数学等价（差一个 log_softmax）|
| **Focal Loss** | 类别极不平衡（如目标检测）| CE × $(1-\hat p_y)^\gamma$，降低易分样本权重 |
| **Label Smoothing CE** | 防止过拟合、提升泛化 | CE 但 one-hot 改成软分布 $(1-\epsilon, \epsilon/(K-1), \ldots)$ |
| **InfoNCE** | 对比学习（CLIP / SimCLR）| 把"正样本对 vs 负样本对"作为分类任务的 CE |
| **Distillation Loss** | 知识蒸馏 | 学生模型对老师 softmax 输出做 CE |
| **Language Model Loss** | GPT / BERT 训练 | 每个位置的下一个 token 是一个 CE 分类问题 |

**InfoNCE 是最重要的一条**——所有现代对比学习（SimCLR、CLIP、SimSiam、MoCo）的损失本质都是**伪装成对比学习的交叉熵**。CE 的统治力远超你想象。

### 收获（背下来）

- **CE = 负对数似然 = KL 散度（差一个常数）= 用错误分布编码真实分布的额外比特**——三种视角通向同一公式，这是它"数学自然"的硬证据，MSE / hinge 都没这种多视角统一性
- **CE 的核心形式**：one-hot 标签下化简为 $-\log \hat p_{y_\text{true}}$，**只看模型给正确类分配了多少概率，取负对数**——简单到不可思议
- **softmax + CE 的梯度恰是 $\hat p - y$**：所有 softmax 雅可比和 log 求导的复杂项**完美抵消**，剩一个减法；这就是你 §第 2 周 `dZ2 = A2 - Y` 的根本来源——不是巧合化简，是数学配对完美
- **CE 对"自信犯错"的惩罚是对数级无界**：$\hat p_y \to 0$ 时 loss $\to \infty$，迫使模型立刻修正最自信的错误；这是它训练效率高于 MSE / hinge 的核心原因
- **CE 的梯度对 logits 有界 $\le \sqrt{2}$**（§Q9 已证）：分类任务配大 lr 不炸的根本是 CE 在兜底；这是 §Q4/§Q9/§Q10 因果链在分类任务上的最终答案
- **CE 永远到不了 0**：$\hat p_y < 1$ 时 $-\log \hat p_y > 0$，训练后期仍有梯度推模型"再自信一点"——这和 hinge 能精确归零形成鲜明对比，也是为什么 CE 必须搭配正则（L2/dropout/early stopping）防过拟合
- **softmax + CE 是数学上的"配对完美"组合**：单独用 softmax + MSE 梯度形式很丑，单独用 sigmoid + MSE 在分类上经常梯度消失（sigmoid 饱和时导数趋 0）；只有 softmax + CE 把所有数学复杂性消干净
- **CE vs hinge 的哲学差别**：CE 关心整个概率分布的形状（概率视角），hinge 关心决策边界两侧的间隔（几何视角）；CE 能给置信度，hinge 不能
- **log-sum-exp trick 是 CE 的数值稳定核心**：永远用框架的 `F.cross_entropy(logits, y)`，不要手写 `log(softmax(...))`——前者内部做 max 减法防溢出
- **`nn.CrossEntropyLoss` 吃 logits 不吃概率**：传 softmax 后的输出会让框架重复做 log_softmax，结果错误且不报错；这是新手最容易踩的接口坑
- **CE 是现代 DL 的隐形统治者**：图像分类（ResNet）、语言模型（GPT 的 token 预测）、对比学习（CLIP/InfoNCE）、知识蒸馏、目标检测（Focal Loss）——背后都是 CE 的变种。学透 CE 等于拿到 90% 现代 DL 模型损失函数的钥匙
- **§Q4/§Q9/§Q10/§Q11 四题的统一论点**：loss 的梯度结构决定优化稳定性，CE 因其概率本质 + 完美梯度化简 + 自然数值稳定，是分类任务在数学上的"标准答案"——这不是约定俗成，是数学硬性结果
- **下次遇到分类任务**：默认 softmax + CE，除非有明确理由（极端不平衡用 Focal、要 margin 控制用 hinge、要软标签用 label smoothing），不要从 MSE 起步——这是教训写在血里的 best practice
